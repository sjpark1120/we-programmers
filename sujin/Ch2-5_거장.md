# 2부 거장

## 2장 배비지: 최초의 프로그래머

- 찰스 배비지
  - 다양한 분야에 관심을 가진 최초의 컴퓨터 엔지니어.
  - 계산표 제작 과정에서 초월함수를 다항식으로 근사하고, 유한 차분법을 활용해 계산을 수행 → 이를 기계로 자동화하려고 ‘차분기관’을 고안함 (결과적으로 차분기관은 완성되지 못함)

- 에이다 러브레이스 백작부인
  - 배비지의 차분기관 개념을 확장해 알고리즘을 작성한 인물로, 최초의 프로그래머로 평가됨

- 느낀점
  - 시도 자체는 혁신적이었지만 끝까지 완성하지 못한 점은 아쉬움
  - 그래도 최초로 계산을 기계화하려 한 시도라는 점에서 의미가 큼

## 3장 힐베르트, 튜링, 그리고 폰 노이만: 최초의 컴퓨터 아키텍트

- 다비트 힐베르트
  - 수학을 완전한 공리 체계로 만들고자 함
  - 하지만 괴델의 불완전성 정리로 인해  
    “참이지만 증명할 수 없는 명제”가 존재함이 밝혀짐 → 수학 체계의 한계 드러남

- 존 폰 노이만
  - 1900년대 헝가리 부다페스트의 부유한 가정에서 태어남
  - 수학적 정밀함을 이용하여 양자 입자의 운명을 결정짓는 숨겨진 변수가 존재하지 않는다는 것을 증명
  - 양자상태의 중첩현상은 극소세게에만 국한되는게 아니라 거시적 존재에도 적용된다라고 주장

- 튜링 폰 노이만 아키텍쳐
  - 기존의 계산 기계는 데이터와 명령어를 분리해서 저장  
    (배비지의 해석기관, 하버드 마크1, SSEC 등)
  - 데이터는 기계 내부에 저장. 명령어는 외부 매체에 저장
  - 두사람은 다른 방식으로 내장식 컴퓨터를 고안

  - 튜링 머신
    - 임의의 수학적 명제가 증명 가능한지 일반적으로 판단하는 방법은 존재하지 않음을 증명
    - 무한한 테이프(메모리)와 상태 전이 규칙으로 구성된 추상 기계
    - 끝없이 긴 종이테이프와 읽기/쓰기 장치를 통해 계산 수행
    - 인간이 직접 조작하는 개념의 중앙 처리 역할 포함
    - 방대한 상태 전이표 -> 서브루틴 개념의 기초
    - 괴델 넘버링: 모든 것을 숫자로 표현
    - 특정 프로그램(sd)을 저장하고, 이를 실행하는 또 다른 프로그램(u)을 구성 가능  
      -> u가 기계화되어 인간이 아닌 자동 기계로 실행된다면 범용 계산기계(Universal Machine)가 됨

  - 탄도 연구소
    - 포탄 궤적 계산은 매우 복잡하고 많은 연산 필요
    - 이를 위해 수많은 여성 계산자(“컴퓨터”)들이 수작업으로 계산 수행
    - 이후 맨해튼 프로젝트 등과 맞물려 계산 자동화 필요성 증가
    - 마크1, ENIAC 등장

  - ENIAC
    - 마크1보다 약 1000배 빠른 성능
    - 하지만 플러그보드 방식으로 프로그램을 설정 → 유연성 부족
    - 종이테이프 방식도 속도 문제로 한계 존재

    - 계산 성능 향상을 위해서는  
      명령어와 데이터를 프로세서와 동일한 속도로 접근 가능한 메모리에 저장해야 함
    - 프로그램을 외부가 아니라 메모리 내부에 저장
    - 프로그램도 데이터처럼 취급 > 현대 컴퓨터 구조의 핵심 개념

- ENIAC 설계 초안
  - 프로그램을 메모리에 저장하는 오늘날의 내장형 프로그램 구조 제안
  - 컴퓨터를 다음 5가지 구성요소로 정의:
    - 입력 / 출력 / 연산 / 제어 / 기억장치
  - 제어장치는 메모리에 저장된 명령어를 해석하고 실행

- 느낀점
  “프로그램도 데이터다”라는 개념이 지금까지 이어진다는 점이 흥미로웠음

## 4장 그레이스 호퍼: 최초의 소프트웨어 엔지니어

- 그레이스 호퍼
  - 자동 프로그래밍 개념 제시  
    → 추상적인 언어로 작성한 프로그램을 컴퓨터가 기계어로 변환  
    → 컴파일러 개념의 시작
  - 프로그래밍을 하나의 체계적인 분야로 정립 → 최초의 소프트웨어 엔지니어

  - 주요 개념 정립
    - 주석, 서브루틴, 디버깅, 컴파일러, 개발 방법론 등
    - 파일, 필드, 파라미터, 루프, 오버플로우, 부동소수점 등 핵심 용어/개념 정립

- 하버드 마크 1
  - 미국 최초의 컴퓨터, 호퍼는 부책임자이자 프로그래머로 참여
  - 레지스터 72개, 각 23자리 십진수 저장
  - 종이테이프(24비트)로 명령 입력
  - 분기/반복문 없음 → 매우 제한적인 구조
  - 직접 구멍을 뚫어 프로그램 작성해야 했음
  - 연산 속도 느림 (로그, 삼각함수 등 특히 느림)
  - 며칠~몇 주 단위 실행
  - 연산 방식도 특이함 (곱셈 결과 소수점 자리만큼 오른쪽으로 이동, 뺄셈은 9의 보수를 더하는 방식)
  - 최적화 기법과 운영 요령 정리 → 일종의 “규율” 형성
  - 출력 형식 개선
  - 파이프라이닝, 멀티프로세싱 개념 도입
  - 점점 시스템이 복잡해짐

- 서브루틴 개념
  - 자주 사용하는 코드 조각을 따로 보관 → 재사용
  - 루틴 라이브러리 구축
  - 상대 주소 개념 도입 → 재배치 가능한 구조
  - 개발 시간 단축

  - 재배치가능한 서브루틴 라이브러리 구현

- 전자식 컴퓨터 등장
  - 진공관 기반 컴퓨터 → 마크1보다 약 5000배 빠름
  - 문제: 초기 설정(프로그래밍)이 너무 오래 걸림
  - 해결 방향:
    - 프로그램을 고속 메모리에 저장해야 함

  -UNIVAC
  - 프로그램 내장형 컴퓨터
  - 수은 지연선 메모리 사용 (40도 이상 온도 유지 필요)
  - 프로그램 흐름이 복잡 → 순서도 필요
  - 간접 주소 미지원 → 자기 수정 코드 사용 시 복잡도 증가

- 정렬, 최초의 컴파일러
  - 병합 정렬 프로그램 등장
  - 정렬 조건(필드 위치, 크기 등)을 입력하면 프로그램 생성
  - “프로그램을 만드는 프로그램” 등장 = 컴파일러

- 알코올

- 컴파일러
  - 초기 프로그래밍은 기계어 수준이라 매우 어려움
  - “컴퓨터가 스스로 프로그램을 생성할 수 없을까?”라는 발상 등장

  - 자동 프로그래밍
    - 고급언어로 작성한 코드를 컴퓨터가 실행 가능한 기계어로 변환
    - 컴파일러의 시작

  - 컴파일러 구조 개념

- a형 컴파일러
  - 프로그래밍 속도를 크게 향상시킴

  - 한계
    1. A-0가 생성한 프로그램은 순수 C-10 코드보다 성능이 떨어짐
    2. 프로그래머들이 컴파일러로 인해 일자리를 잃을 것이라는 불안감 존재

- 언어
  - 대수적(수학적) 컴파일러 등장
  - 오버레이 기법, 코어 메모리 등 기술 발전
  - 하지만 여전히 일반 사용자가 쓰기에는 어려움 존재

  - 결론
    - 사람들이 더 쉽게 사용할 수 있는 고수준 언어 필요

- COBOL
  - 기존 대수적 언어는 과학자/수학자 중심 → 비즈니스 사용자에게는 어려움
  - 영어 기반의 직관적인 언어를 목표로 개발
    - B-0 언어 개발 착수
    - 공통으로 사용할 수 있는 표준 언어 필요성 제기

  - 결과
    - 공통 업무 지향 언어 COBOL 탄생
    - 실제로 2000년대까지 작성된 프로그램의 80% 이상이 COBOL 기반

- 느낀점
  컴파일러 개념이 처음 나왔을 때도 성능 문제나 개발자 반발이 있었다는 점이 흥미로웠음  
  → 지금 프레임워크나 자동화 도구(AI)에 대한 반응이랑 크게 다르지 않다고 느낌

- 결국 기술은 성능보다 “사람이 쓰기 쉬운 방향”으로 진화한다는걸 느낌
